    【设计模式】 相关知识

    寒铁 2021-02-26 java

    常用设计思路：

    1. 定义接口类，定义方法
    2. 定义抽象类实现接口类，定义模板方法，且实现一些通用操作，作代码复用
    3. 实现常用的应用类继承抽象类，作基础开发使用
    4. 后续扩展实现应用类或抽象类

    即模板模式，重点是抽象类，可多重抽象，主要规划逻辑操作执行的流程顺序！

    # 设计原则

    原则 含义 具体方法
    开闭原则 对扩展开放,对修改关闭 多使用抽象类和接口
    里氏代换原则 基类可以被子类替换 使用抽象类继承,不使用具体类继承
    依赖倒转原则 要依赖于抽象,不要依赖于具体 针对接口编程,不针对实现编程
    接口隔离原则 使用多个隔离的接口,比使用单个接口好 建立最小的接口
    迪米特法则 一个软件实体尽可能少地与其他实体发生相互作用 通过中间类建立联系
    合成复用原则 尽量使用合成/聚合,而不是使用继承 尽量使用合成/聚合,而不是使用继承

    # 设计模式分类

    # 创建型模式

    侧重于创建对象，隐藏创建逻辑，使得创建对象更加灵活

    • 工厂模式（Factory Pattern）
    • 抽象工厂模式（Abstract Factory Pattern）
    • 单例模式（Singleton Pattern）
    • 建造者模式（Builder Pattern）
    • 原型模式（Prototype Pattern）

    # 结构型模式

    共8种，侧重于通过继承的概念组合对象获得新功能

    • 适配器模式（Adapter Pattern）
    • 桥接模式（Bridge Pattern）
    • 过滤器模式（Filter、Criteria Pattern）
    • 组合模式（Composite Pattern）
    • 装饰器模式（Decorator Pattern）
    • 外观模式（Facade Pattern）
    • 享元模式（Flyweight Pattern）
    • 代理模式（Proxy Pattern）

    # 行为型模式

    共12种，侧重于对象之间的通信

    • 责任链模式（Chain of Responsibility Pattern）
    • 命令模式（Command Pattern）
    • 解释器模式（Interpreter Pattern）
    • 迭代器模式（Iterator Pattern）
    • 中介者模式（Mediator Pattern）
    • 备忘录模式（Memento Pattern）
    • 观察者模式（Observer Pattern）
    • 状态模式（State Pattern）
    • 空对象模式（Null Object Pattern）
    • 策略模式（Strategy Pattern）
    • 模板模式（Template Pattern）
    • 访问者模式（Visitor Pattern）

    # 创建型模式

    # 工厂模式、抽象工厂模式

    • 只需提供类型，由工厂对象判断类型统一生成，对象的功能接口由自己实现
    • 作用：简化创建对象时的选择
    • 抽象工厂，多封装一层工厂抽象类，不同类型工厂对象也根据父级抽奖类统一生成
    • 典型例子：一个超级工厂抽象类
      • 形状工厂，获取圆形、矩形等对象，这些对象属于同一系列，有公共接口，具体实现由具体形状决定
      • 颜色工厂，获取不同颜色对象，这些对象属于同一系列，有公共接口，具体实现由具体颜色决定

    # 单例模式

    • 保证一个类只有一个实例，避免反复创建销毁
    • 适用：全局使用的类
    • 6种方式
      • 懒汉，线程不安全，懒加载
      • 懒汉，线程安全（加锁），懒加载
      • 饿汉，线程安全（类装载初始化），非懒加载
      • 静态内部类，线程安全（类装载不初始化，但静态固定修饰不可变），懒加载
      • 双校验锁，线程安全（双重锁），懒加载
      • 枚举，线程安全（唯一），懒加载，且自动序列化
    【示例】 
    /**
 * 懒汉式，线程不安全，不推荐
 */
class Singleton {
    private static Singleton instance;

    private Singleton() {
    }

    public static Singleton getInstance() {
        if (instance == null) {
            instance = new Singleton();
        }
        return instance;
    }
}

/**
 * 懒汉式，线程安全，不推荐
 */
class Singleton2 {
    private static Singleton2 instance;

    private Singleton2() {
    }

    // 加同步锁
    public static synchronized Singleton2 getInstance() {
        if (instance == null) {
            instance = new Singleton2();
        }
        return instance;
    }
}

/**
 * 饿汉式，线程安全
 */
class Singleton3 {
    /** 没有懒加载效果，即JVM装载类直接初始化，和内存有关（类属性都不推荐直接给初始值，尤其POJO类）**/
    private static Singleton3 instance = new Singleton3();
    private Singleton3 (){}
    public static Singleton3 getInstance() {
        return instance;
    }
}

/**
 * 双重校验锁，线程安全
 */
class Singleton4 {
    private volatile static Singleton4 singleton;
    private Singleton4 (){}
    public static Singleton4 getSingleton() {
        if (singleton == null) {
            synchronized (Singleton4.class) {
                if (singleton == null) {
                    singleton = new Singleton4();
                }
            }
        }
        return singleton;
    }
}

/**
 * 静态内部类，线程安全，且懒加载
 */
class Singleton5 {
    private static class SingletonHolder {
        private static final Singleton5 INSTANCE = new Singleton5();
    }
    private Singleton5 (){}
    public static Singleton5 getInstance() {
        return SingletonHolder.INSTANCE;
    }
}

/**
 * 枚举方式
 * 优点：简洁、枚举的自动序列化机制不用担心反序列化
 */
enum Singleton6 {
    INSTANCE;
}

    # 享元模式（Flyweight Pattern）

    • 提供共享对象来使用，避免重复创建而占用内存。

    • 场景例子：缓存池（数据库连接池、线程池、String的字符串缓存池）

    • 代码例子：

      【示例】基于工厂模式的例子修改 
      class FwShapeFactory {
    /** 缓存池 */
    private static final HashMap<String, Shape> circleMap = new HashMap<>();

    public static Shape getFwCircle(String color) {
        FwCircle circle = (FwCircle) circleMap.get(color);
        // 重复对象缓存，避免重复创建
        if (circle == null) {
            circle = new FwCircle(color);
            circleMap.put(color, circle);
            System.out.println("Creating circle of color : " + color);
        }
        return circle;
    }
}

    # 适配器模式（Adapter Pattern）

    • 适配器继承或依赖已有的对象，实现想要的目标接口
    【示例】 
    /**
 * 例子：音频播放器，适配不同格式的文件类型，原功能没有的接口，由适配器对象实现
 * 实现方式：依赖，（不推荐继承，因为类的单一继承性）
 */
public class AudioPlayer implements MediaPlayer {
    MediaAdapter mediaAdapter;

    @Override
    public void play(String audioType, String fileName) {

        //播放 mp3 音乐文件的内置支持
        if (audioType.equalsIgnoreCase("mp3")) {
            System.out.println("Playing mp3 file. Name: " + fileName);
        }
        //mediaAdapter 提供了播放其他文件格式的支持
        else if (audioType.equalsIgnoreCase("vlc")
                || audioType.equalsIgnoreCase("mp4")) {
            mediaAdapter = new MediaAdapter(audioType);
            mediaAdapter.play(audioType, fileName);
        } else {
            System.out.println("Invalid media. " + audioType + " format not supported");
        }
    }
}

    # 装饰器模式（Decorator Pattern）

    • 扩展类方法

    • 常规方式：子类继承，重写方法。导致子类更多。

    • 方式：定义抽象装饰类实现接口方法，指定装饰对象

    【示例】 
    public class RedShapeDecorator extends ShapeDecorator {

   //  装饰对象
   public RedShapeDecorator(Shape decoratedShape) {
       super(decoratedShape);     
   }

   // 接口方法，
   @Override
   public void draw() {
       decoratedShape.draw();  
       // 扩展的内容
       setRedBorder(decoratedShape);
   }


   private void setRedBorder(Shape decoratedShape){
       System.out.println("Border Color: Red");
   }
}

    # 外观模式（Facade Pattern）

    • 即门面模式
    • 场景：SLF4J 日志框架，具体由Log4j 、Logback等实现
    • 作为中间层，避免客户端与系统直接耦合，有利于框架变更而无感知

    # 观察者模式（Observer Pattern）

    • 一个对象发生改变，依赖于他的对象收到通知自动更新
    • 一对多：被观察者/观察者，被监听者/监听者，发布/订阅
    • 被观察者：集合属性，用于存储观察者
    • 观察者：定义观察接口
    • 应用：
      • 如 java.util.EventListener
      • jdk：java.util.Observable被观察者和java.util.Observer观察者接口
      • spring：事件监听器机制
        • ApplicationEvent事件抽象类
        • org.springframework.context.ApplicationListener<E extends ApplicationEvent>监听器接口，实现类要注入容器
        • org.springframework.context.ApplicationContext上下文发布事件
      • 第三方消息中间件

    # 模板模式（Template Pattern）

    • 抽象父类，子类必须实现各自的逻辑；父类通用接口，一般制定逻辑操作的流程，每个操作是一个抽象模板接口
    【示例】 
    /**
 * 抽象父类，通用方法定义逻辑执行的流程，每个接口为模板接口
 */
abstract class Game {
    abstract void initialize();

    abstract void startPlay();

    abstract void endPlay();

    /** final修饰，不可变 */
    public final void play() {

        if (this.isSupported()) {
            //初始化游戏
            initialize();

            //开始游戏
            startPlay();

            //结束游戏
            endPlay();
        } else {
            System.out.println("游戏不支持！");
        }
    }

    abstract boolean isSupported();
}

    # 策略模式（Strategy Pattern）

    • 具体策略对象，实现统一的策略接口，具体策略算法由具体类封装，独自维护
    • 应用类，可继承或依赖方式集成，推荐依赖方式
    【示例】 
    /**
 * 应用类，用策略属性执行策略
 */
class Context {
    private Strategy strategy;

    public Context(Strategy strategy) {
        this.strategy = strategy;
    }

    public int executeStrategy(int num1, int num2) {
        return strategy.doOperation(num1, num2);
    }
}

/**
 * 策略接口
 */
interface Strategy {
    public int doOperation(int num1, int num2);
}

class OperationAdd implements Strategy {
    @Override
    public int doOperation(int num1, int num2) {
        return num1 + num2;
    }
}

class OperationSubtract implements Strategy {
    @Override
    public int doOperation(int num1, int num2) {
        return num1 - num2;
    }
}

    # 责任链模式（Chain of Responsibility Pattern）

    • 每个对象都可以传递下个对象，形成链式请求
    • 适合多层过滤请求
    • 例子：过滤器链
    【示例】 
    public abstract class AbstractLogger {
    public static int INFO = 1;
    public static int DEBUG = 2;
    public static int ERROR = 3;

    protected int level;

    /** 责任链中的下一个元素 */
    protected AbstractLogger nextLogger;

    public void setNextLogger(AbstractLogger nextLogger) {
        this.nextLogger = nextLogger;
    }

    /**
     * 父类通用接口，传递请求
     * @param level
     * @param message
     */
    public void logMessage(int level, String message) {
        // 先执行下个节点，即先进后出
        if (this.level <= level) {
            write(message);
        }
        if (nextLogger != null) {
            nextLogger.logMessage(level, message);
        }
        // 后执行下个节点，即先进先出
//        if (this.level <= level) {
//            write(message);
//        }
    }

    /**
     * 实际操作接口
     * @param message
     */
    abstract protected void write(String message);

}

    # 中介者模式（Mediator Pattern）

    • 非显现调用对象, 降低对象之间的直接引用关系
    • 具体类中都封装一个统一中介对象
    • 多对一的关系,简化为一对一
    • 扩展,思考函数式接口Supplier<T>
    【示例】 
    public class MediatorPattern {
    public static void main(String[] args) {
        User robert = new User("Robert");
        User john = new User("John");

        robert.sendMessage("Hi! John!");
        john.sendMessage("Hello! Robert!");
    }
}

/**
 * 中介者，实际操作对象，但不直接显现
 */
class ChatRoom {
    public static void showMessage(User user, String message) {
        System.out.println(new Date().toString()
                + " [" + user.getName() + "] : " + message);
    }
}

class User {
    private String name;

    public String getName() {
        return name;
    }

    public void setName(String name) {
        this.name = name;
    }

    public User(String name) {
        this.name = name;
    }

    /**
     * 隐藏实际的调用对象，简化对象引用关系
     */
    public void sendMessage(String message) {
        ChatRoom.showMessage(this, message);
    }
}
