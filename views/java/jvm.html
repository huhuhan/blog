    # JVM 简介

    JVM 即 JAVA 虚拟机 ，由C语言开发。

    JVM也有不同厂商开发，主要有如下几种

    • HotSpot，即最常见，Sun 公司开发。后续内容就是这个版本JVM
    • JRockit，是Oracle 公司开发
    • IBM，即IBM公司开发

    简单例子介绍

    java demo.class
    1
    1. 运行以上命令，操作系统创建一个进程执行JAVA程序
    2. JAVA程序会创建JVM，JVM加载运行
    3. JAVA通过JNI调用C语言代码实现Socket通信，JVM处理字节码命令
    4. 进程的虚拟地址空间，分配一部分空间给JVM使用，即JVM内存
    5. 进程的剩余其他空间，即进程内存，对JVM来说，统称为本地内存

    参考：博客 (opens new window)、博客 (opens new window)

    # 内存结构

    # 运行时数据区

    • 堆内存：理解为管理内存的主要区域，多线程共享。

      1. 存储的数据包括
	1. 对象实例（Student对象）
	2. 数组（数据，非引用）
2. 垃圾回收负责的主要区域
3. 逃逸分析（待研究）

    • 方法区：和堆一样，多线程共享的内存区域。

      1. 存储的数据包括
	1. 虚拟机加载的类（System类）
	2. 静态变量（static修饰的Student类）
	3. 常量（NUMBER，在常量池中）
	4. 即时编译后的class代码（比如动态代理生成的代理类）
	5、全局对象的引用（stu）？？
2. 常量池，独立的内存区域，JDK6前在方法区中，JDK7后在堆内存中
3. 永久代，运行时数据区的5个区域理解为是JVM的设计规范，永久代是HotSpot虚拟机对方法区的实现方案，JDK7后又取消永久代的设计。
4. 元空间，JDK8新加，方法区新的实现方案，不过存储位置在本地内存中，不在JVM内存中

    • 程序计数器：理解为存储当前线程的指令步骤。线程私有

      因为一个CPU在某一时刻只能处理一个线程的一个指令，多线程情况下，CUP需要切换线程处理。保证切换后回到各自线程继续处理指令。运行java方法，存储字节码指令地址；运行native方法，不存储；
      1

    • 虚拟机栈：理解为Java方法运行的内存模型。线程私有

	3. 动态链接（指向常量池的引用）
	4. 方法返回地址（当前栈帧的出栈，将返回值给上层方法栈帧，继续处理）
	5. 其他信息

    # 垃圾回收机制

    JAVA GC 即Garbage Collection， 垃圾回收机制，是JVM内存回收机制，简称GC。

    运行时数据区的5大区域中，主要针对共享区的堆内存、方法区，线程私有的内存随着线程销毁回收。

    # 判断存活算法

    GC回收堆内存、方法区中的对象，通过判断对象是否存活（死去即不再被引用）

    引用计数算法
早期算法，这种算法判断很简单，就是给对象添加一个引用计数器，每当对象被引用一次就加1，引用失效时就减1。当为0的时候就判断对象不会再被引用。
优点:实现简单效率高，被广泛使用与如python何游戏脚本语言上。
缺点:难以解决循环引用的问题，就是假如两个对象互相引用已经不会再被其它其它引用，导致一直不会为0就无法进行回收。
    1
    2
    3
    4

    可达性分析算法：通过一个称为“GC Roots”的对象为起始点，搜索所经过的路径称为引用链，当一个对象到GC Roots没有任何引用跟它连接，则证明对象是不可用的。有效解决了引用计数算法的循环利用弊端。

    可作为GC Roots的对象有四种

    1. 虚拟机栈(栈桢中的本地变量表)中的引用的对象。
2. 方法区中的类静态属性引用的对象，一般指被static修饰引用的对象，加载类的时候就加载到内存中。
3. 方法区中的常量引用的对象,
4. 本地方法栈中JNI（native方法)引用的对象

    # 垃圾回收算法

    为了高效的回收，jvm将堆分为三个区域
1. 新生代（Young Generation）NewSize和MaxNewSize分别可以控制新生代的初始大小和最大的大小
2. 老年代（Old Generation）
3. 永久代（Permanent Generation）【1.8以后采用元空间Metaspace，就不在堆中了】
    1
    2
    3
    4

    ​

    JVM采用分代收集算法对不同区域、不同大小对象、不同对象存活时长，采用不同的回收算法。

    新生代，采用复制算法。大部分对象存活率低，少量对象的区域复制就完成清除。

    • 优先在Eden区分配，其次From Survivor区，To Survivor区为空。（两个Survivor区，简称From 区和To 区）
      • 当Eden区不够空间分配，发起一次Minor GC
      • Eden区存活对象复制到To 区
      • From区，根据存活年龄，分到To 区或老年代

    老年代，采用标记/整理或标记/清除算法

    • 新生代复制而来的存活对象

    • 直接进入老年代的大对象（避免内存复制）

    • 老年代不足空间，进行Full GC回收。（优化之一就是尽量避免触发Full GC）

    • 通过对存活对象的标记，整理或直接清除，得到连续的空闲内存

    # 垃圾回收器

    即垃圾回收的具体实现，针对不同场景和配置，GC采用不同的回收器，可配合使用，如图

    新生代
Serial：单线程操作，回收时停止其他线程，结束后其他线程再工作。简单高效。采用复制算法。
ParNew：多线程操作，Serial升级版，可以配合CMS回收器一起用。
Parallel Scavenge：多线程操作，侧重吞吐量
【吞吐量 = 代码运行时间 / (代码运行时间 + 垃圾收集时间) 如果代码运行100min垃圾收集1min，则为99%】

老年代，
Serial Old：单线程操作，类似Serial，但算法不同（标记-整理算法）
Parallel Old：多线程操作
CMS：支持并发，采用标记-清除算法，生命周期（初始标记-并发标记-重新标记-并发清除）
    1

    最新G1回收器
1. JDK1.7出现，理解为CMS的升级版，解决CMD产生的空间碎片（压缩空间的功能）。
2. 算法是标记-整理 + 复制。
3. 混合使用，不属于某分区。
4. 默认关闭（和JDK版本有关），开启配置【-XX:+UseG1GC】，
5. 将堆内存区域化，标记为若干个【region】，配置参数【-XX:G1HeapRegionSize=n】默认自动分配（2的幂）
6. 采用并发标记、并发复制操作，降低停顿时间。最大停顿时间【-XX:MaxGCPauseMillis=200】默认200

    回收模式
1. Minor GC: 在新生代(包括Eden区和Survivor区)中的垃圾回收。默认经过15次后还存活就复制到老年代。
2. Major GC: 在老年代的垃圾回收，有时需要将新生代清理将部分对象复制而来，类似Full GC
3. Full GC: 
	1. 所有区域的回收，包括新生代、老年代、元空间等等。
	2. 【System.gc()】禁止直接使用。
	3. 老年代空间不足会触发。优化目标就是避免触发。
4. Mixed GC: 仅G1回收器才有。回收新生代和部分老年代

    # 疑问点

    • 标记算法中，如何判断对象是否被标记？

      通过JVM提供的OOPMap数据结构，存储引用链上引用过的位置，避免全局扫描
      1

    • 什么是空间分配担保？

      在minor gc前，jvm会先检查老年代最大可用空间是否大于新生代所有对象总空间，如果是的话，则minor gc可以确保是安全的
      1

    • 为什么新生代的复制算法分2个Survivor区？

      保证复制算法运行的效率，直接复制去老年代可能导致内存不足，直接触发Full GC。且老年代的对象是经过多次可达性分析算法存活的对象。（默认15次）
      1

    • 为什么是Stop the World？

      进行GC操作时，停止除GC外的所有线程。（侧重高吞吐的回收器，需要减短停顿时间）
      1

    # 调优分析

    # 配置参数

    • -： 标准VM选项，VM规范的选项。
    • -X： 非标准VM选项，不保证所有VM支持。
    • -XX： 高级选项，高级特性，但属于不稳定的选项。

    老年代需要计算，空间大小 = 堆空间大小 - 新生代大空间大小

    常见参数：

    • -Xms设置堆的最小空间大小。
    • -Xmx设置堆的最大空间大小。
    • -Xmn堆中新生代初始及最大大小（NewSize和MaxNewSize为其细化）。
    • -XX:NewSize设置新生代最小空间大小。
    • -XX:MaxNewSize设置新生代最大空间大小。
    • -XX:PermSize设置永久代最小空间大小。（jdk1.8后permSize替换为metaspaceSize）
    • -XX:MaxPermSize设置永久代最大空间大小。（jdk1.8后permSize替换为MaxMetaspaceSize）
    • -Xss设置每个线程的堆栈大小。

    其他参数

    【-XX:+PrintGCDetails】 打印GC日志
【-XX:+PrintGCDateStamps】 GC日志加上时间戳
【-XX:+PrintGCTimeStamps】 GC每次输出日志的计时（从0开始）
【-Xloggc:./yh_gc.log】 GC日志文件路径（当前路径）
【-XX:UseParNewGC】使用ParNew垃圾回收器，其他参考垃圾回收器的种类
【-XX:UseG1GC】 使用G1垃圾回收器，适合堆内存2G起
【-XX:MaxGCPauseMillis】 G1垃圾回收器的停顿时长，毫秒
【-XX:+ParallelRefProcEnabled】 开启并行处理引用对象
【-XX:+HeapDumpOnOutOfMemoryError】 发成OOM时生成DUMP文件
【-XX:HeapDumpPath】 DUMP文件路径

    # 内存溢出

    OOM，即OutOfMemeyError，常见以下几种情况：

     Exception in thread “main”: java.lang.OutOfMemoryError: Java heap space
    1
    1
     Exception in thread “main”: java.lang.OutOfMemoryError: PermGen space
    1
    1
    • 虚拟机栈、本地方法栈：
      • 存在多线程时，栈帧空间不够分配，抛出OOM
      • 仅单线程时，栈帧空间不够分配，抛出StackOverflowError

    常见的OutOfMemeyError类型如下：

    • 对象不能被分配到堆内存中

      Exception in thread “main”: java.lang.OutOfMemoryError: Java heap space
      1

    • 类或者方法不能被加载到老年代。它可能出现在一个程序加载很多类的时候，比如引用了很多第三方的库。

      Exception in thread “main”: java.lang.OutOfMemoryError: PermGen space
      1

    • 创建的数组大于堆内存的空间

      Exception in thread “main”: java.lang.OutOfMemoryError: Requested array size exceeds VM limit
      1

    • 分配本地分配失败。JNI、本地库或者Java虚拟机都会从本地堆中分配内存空间。

      Exception in thread “main”: java.lang.OutOfMemoryError: request <size> bytes for <reason>. Out of swap space?
      1

    • 同样是本地方法内存分配失败，只不过是JNI或者本地方法或者Java虚拟机发现。

      Exception in thread “main”: java.lang.OutOfMemoryError: <reason> <stack trace>（Native method）
      1
