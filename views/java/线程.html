# 线程

# 线程的创建

  • 继承Thread类
  • 实现Runnable接口

执行start方法，创建线程，相对整体来说多了一条线程执行，体现多线程的性质 执行run方法，运行线程的代码，相对该线程本身，内部代码执行是同步的。

# 线程状态

共六种：NEW（创建）、RUNNABLE（运行）、BLOCKED（阻塞）、WAITING（等待）、TIMED_WAITING（定时等待）、TERMINATED（终止）

# 线程安全

在多线程下执行和单线程下执行的结果永远一致，即线程安全。线程安全其本质，就是存储在内存中的数据安全，存在堆中数据是多线程共享，涉及安全问题。

线程安全方案分类：

  • 位置隔离：数据存储在线程私有内存，即栈内存，比如局部变量
  • 数据隔离：数据存储在公共堆内存，但保证每个线程各自用各自的数据，比如ThreadLocal
  • 只读数据：数据存储在公共堆内存，但标记只读，不可修改，比如final修改的常量（常量池在堆内存）
  • 加锁数据：数据存储在公共堆内存，使用时加锁，保证不被其他线程使用，用完释放锁，比如互斥锁
  • 乐观数据：数据存储在公共堆内存，并发概率小的情况，CAS（Compare And Swap），用完记录下值，下次使用比较是否变化，没有变化则安全（可能变了版本，值不变）否则重新再来；或者直接不加锁

# 本地线程

即ThreadLocal，非线程，而是线程的局部变量，通过维护本地线程的变量，保证这些变量使用时线程安全，不同线程存储各自的变量，互不影响，即数据隔离；

场景：

  • 用户登录的认证信息，如Spring Security的SecurityContextHolderStrategy接口实现类
  • 动态数据库的当前数据库连接，如MyBatis-Plus的DynamicDataSourceContextHolder
  • 事务隔离级别，如spring的事务隔离TransactionSynchronizationManager
  • 全局时间格式化对象

原理：

  • ThreadLocal类里有一个ThreadLocalMap内部类，其中又定义了Entry内部类，这类继承了WeakReference弱引用类

  • 每次操作都是从当前线程获取的，即数据存储在各自线程的ThreadLocalMap

    public void set(T value) {
      Thread t = Thread.currentThread();
      // ThreadLocal的set、get方法都先获取ThreadLocalMap
      ThreadLocalMap map = getMap(t);
      if (map != null)
          map.set(this, value);
      else
          createMap(t, value);
  }
  public T get() {
      Thread t = Thread.currentThread();
      // ThreadLocal的set、get方法都先获取ThreadLocalMap
      ThreadLocalMap map = getMap(t);
      if (map != null) {
          ThreadLocalMap.Entry e = map.getEntry(this);
          if (e != null) {
              @SuppressWarnings("unchecked")
              T result = (T)e.value;
              return result;
          }
      }
      return setInitialValue();
  }
  // 重点，从数据来自本地线程Thread对象中
  ThreadLocalMap getMap(Thread t) {
      return t.threadLocals;
  }

  • corePoolSize：核心线程数，线程池中始终存活的线程数
  • maximumPoolSize：最大线程数，线程池中允许的最大线程数，当线程池的任务队列满了之后可以创建的最大线程数
  • keepAliveTime：最大线程数存活的时间，当线程中没有任务执行时，保留一段时间再销毁，数量保持为核心线程数量
  • unit：keepAliveTime的时间单位，共7种，天、小时、分、秒、毫秒、微秒、纳秒
  • workQueue：阻塞队列，用来存储线程池等待执行的任务，均为线程安全
    • ArrayBlockingQueue：基于数组的先进先出队列，有界（必须设置长度）
    • LinkedBlockingQueue：默认，基于链表的先进先出队列，默认构造参数是Integer.Max_Value，等于无界队列，最大线程数失效。因此建议手动设置长度来创建
    • SynchronousQueue：无缓冲的等待队列，即不阻塞，直接交给线程
    • 接口BlockingQueue的其他实现类
  • threadFactory：线程工厂，主要用来创建线程
  • handler：拒绝策略，队列已满且线程数达到最大时的处理策略
    • AbortPolicy：默认丢弃任务，并抛出异常
    • DiscardPolicy：丢弃任务，不抛出异常
    • DiscardOldestPolicy：丢弃队列最前面的任务，加入被拒绝的任务
    • CallerRunsPolicy：（主线程）直接执行该任务
    • 接口RejectedExecutionHandler，实现自定义策略

# 执行流程

ThreadPoolExecutor 执行流程如图

  1. 当线程数小于核心线程数时，创建线程执行任务。

  2. 当线程数大于等于核心线程数，

    • 若任务队列未满，进入阻塞的任务队列，等待被线程调用执行。

    • 若任务队列已满，

      1. 线程数小于最大线程数，创建线程直接执行任务，即没有入队列。
      2. 若线程数等于最大线程数，执行拒绝策略（默认抛出异常）

核心线程数可以理解是项目程序预估需要处理的线程数量，所以优先达到核心线程数，即便是懒加载的形式（先初始化再足量增加）。

最大线程数是处理突发情况而增量的线程数，在线程空闲下来时，会逐渐销毁，数量保持为核心线程数。

【示例】

单元测试不支持多线程的测试，其源码得知主线程结束就关闭进程了，所以用静态main()调用测试。

    public static void main(String[] args) {
        // 创建线程池
        ThreadPoolExecutor threadPool = new ThreadPoolExecutor(
                5,
                10,
                100,
                TimeUnit.SECONDS,
                new LinkedBlockingQueue<>(8)
                // 参数6：默认为正常优先级、非守护线程
                // 参数7：默认AbortPolicy：拒绝并抛出异常
            	// new ThreadPoolExecutor.DiscardPolicy() // 直接忽略不报异常
        );
        // 执行任务
        for (int i = 0; i < 10; i++) {
            final int index = i;
            threadPool.execute(() -> {
                System.out.println(index + " 被执行,线程名:" + Thread.currentThread().getName());
                try {
                    Thread.sleep(1000);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
            });
        }
        // 启动一次顺序关闭：当前线程池已提交的任务，按顺序都执行完毕后，线程池关闭
        threadPool.shutdown();
    }

# 线程池的状态

  • Running（运行）：能接收任务，并处理阻塞队列的任务
  • Shutdown（停工）：不接收任务，但能处理阻塞队列的任务
  • Stop（停止）：不接收任务，不处理阻塞队列任务，还会停止运行任务中的线程
  • Tidying（整理）：所有任务终止，工作线程为0，处于关闭之前的状态
  • Terminated（终止）：已关闭

# 配置参考

常见公式：线程数 = CPU核数 *（1+线程等待时间 / 线程时间运行时间）

可以作为初始化调试的数据，通过不断迭代的调试压测再修改
