                  # 常用命令总结

                  # 基本操作

                  # 克隆、初始化
git clone ...
git init
# 查看、删除文本文件
cat README.md
rm README.md
# 添加版本控制
git add README.md
git add .
# 根据不同状态下撤销
git status
git restore [--staged] <file> ...
# 拉取代码
git pull origin
# 提交、推送
git commit -m "update"
git push origin master

                  # 分支操作

                  # 创建、切换、查看、删除分支
git switch [-c] dev 或 git checkout [-b] dev
# 查看本地分支，[-r]本地的远程分支，[-a]本地的所有分支
git branch 
# 更新本地的远程分支，参考IEAD的fetch命令
git fetch origin --progress --prune
# 切换、合并到当前分支
git switch master
git merge dev
# 存在冲突，当前状态为（master|merging）
git status 
# 手动解决后，再提交
git add .
git commit -m 'merge by master'
git push origin master
# 冲突建议通过第三方可视化比较工具解决

                  # 版本回滚

                  git reset [参数][commit_id]
                  1

                  例子如下

                  # 假设git本地日志如下：commit_id实际是哈希值
提交3："1月3日上传了文件" commit_id=b333333333

# 假设git服务器日志如下：
提交2："1月2日上传了文件" commit_id=b221222222
提交1："1月1日上传了文件" commit_id=b111111111

                  回滚【提交3】，复制【提交2】的版本号，用IDEA的插件窗口LocalChanges

                  工作区：代码版本没变化，IDEA标记白色或红色（还没版本号）

                  缓存区：代码版本变化，如IDEA上标记蓝色或绿色

                  IDEA的版本控制窗口LocalChanges会显示红色、蓝色、绿色的代码

                  • mixed ：会【提交3】代码回滚本地，LocalChanges中原本蓝色还是显示蓝色，原本红色还是红色。适用可以继续修改，再重新提交
                  • solt：类似mixed，区别不大，不多用
                  • hard：等于执行mixed后，有对回滚本地的蓝色代码，再执行Revert命令

                  # 本地仓库回滚

                  如上，一般情况是cmmit注释写错，回滚重新提交；或者需要回滚分几次提交。

                  # 远程仓库回滚

                  一样，根据情况回滚到目标版本。然后推送是用强制推送，覆盖远程仓库记录

                  git push -f
                  1

                  远程仓库设置成被保护的分支，即使允许推送，也不能推送

                  # 注意事项

                  内网部署社区办的gitlab，因为管理员可以有任何权限，因此开发人员不应当同时兼任管理员。 否则，管理员可以参与项目开发，切项目成员权限分配限制不了管理员，容易造成项目开发路线混乱。权限分配失去意义

                  以上情况尤其会出现公司不规范管理，开发和管理没详细分工的情况

                  # 操作逻辑

                  当前服务器默认分支：master，简称S分支

                  1. 更新分支，命令参考（git fetch origin --progress --prune）
                  2. 根据S分支，切出一份本地的分支，S1分支
                  3. 以S1分支，切出自己开发用的分支，简称A1分支(格式推荐：名称+日期，如xxx1027）
                  4. 自己A1分支提交. 推送
                  5. 服务器上发起合并请求，合并请求处理成功
                  6. 执行操作1步骤
                  7. 切到本地S1分支，拉取S分支最新代码
                  8. 以S1分支，再次切出自己开发用的新分支A2
                  9. 继续在A2分支上提交. 推送，后续步骤类似
                  1. 保留2份自己的A分支，当前开发，和上次开发但已提交合并
                  2. 本地S1分支，只做同步服务器S分支用，区分本地自己A分支
                  3. 步骤1，很重要，遗漏的话会导致后续S或S1分支切出的A2分支，没有已合并的A1代码和其他人提交的代码
                  4. 步骤8，可以继续在A1分支上开发，但需要将已同步过的S1分支合并过来，不建议这样操作！ 尤其步骤1没操作即没同步，就需要自己处理合并冲突，生成新的commit记录，再次提交推送合并会影响其他版本代码
