HTTPS 加密访问

寒铁 2022-03-17 network

# HTTPS 应用

# 简介

HTTPS，即超文本传输安全协议，比HTTP协议多了一层，安全套接层（SSL/TSL，SSL标准化后改名为TSL）。

在通信端口上，HTTP默认端口是80，HTTPS默认端口是443

#

# 加密机制

SSL/TSL 这层就负责处理加密解密，HTTPS主要采用数字证书实现。以下依次介绍各种加密方案

# 对称

对称加密算法：加密解密用同一个密钥

优点：长度固定，加解密耗时短

方案缺点：无论是客户端还是服务端生成密钥A，密钥在传输给对方时可能被劫持，那么安全就无法保证了

ps：即使提前存储，对于B/S架构，客户端浏览器不可能存储所有的HTTPS网站的密钥

# 非对称

非对称加密算法：需要一组密钥对，公钥加密的内容需要对应的私钥解密，私钥加密的内容需要对应的公钥解密，可保证单方向传输的安全性。

则可以进阶得出以下方案，两组密钥对

  1. 先传输各自的公钥给另一端
  2. 另一端用公钥加密传输数据，自己这段用私钥解密

方案缺点：

  • 非对称加密算法，解密耗时长，特别是较大数据

# 非对称+对称

  1. 客户端从服务端获取的非对称公钥A1
  2. 客户端随机生成密钥X，并用非对称公钥A1，加密，传输给服务端。保证密钥X不泄露。
  3. 数据传输用密钥X加密。保证加解密性能问题。

但是，第一点有漏洞，比如**中间人劫持，构造假密钥**

问题：如何保证客户端收到的公钥是正确安全的？

# 数字证书（非对称+签名+对称）

针对上面的问题，用数字证书可以保证客户端收到公钥是正确的！

证书由CA机构颁发。（CA机构，即互联网公信机构）

证书信息很多，可以找个网站查看，如下

那么证书，可能会有中间人劫持伪造吗？不会！因为其中有数字签名！

数字证书申请制作步骤：

  1. 填写基本信息，比如地区、组织、名称（域名）等等，去CA机构申请审核
  2. CA机构用基本信息，生成非对称密钥M1，其中M1的公钥也就是证书源文件（CSR文件）
  3. CA机构将基本信息，用哈希算法计算得到哈希值，假设为H1
  4. CA机构将H1用私有的非对称加密，得到签名，即数字签名。（假设这里是M2，另一组密钥）
  5. 将数字签名、证书源文件（M1公钥）、用的哈希算法、有效期、基本信息等等，组成数字证书（也是加密文件）
  6. 最终CA机构返回数字证书文件、M1的私钥等多个文件

服务器配置，只需用到数字证书文件、M1的私钥（绝不可泄露）

所以，即使中间人劫持，没有CA的私钥M2是无法伪造签名的

客户端浏览器验证数字证书中的签名：

  1. 获取数字证书，解析出其中信息（数字签名、证书源-M1公钥、用的哈希算法），对应上面第5点
  2. 根据指定的哈希算法，同理计算得出H2，对称上面第3点
  3. 浏览器会内置信任机构的公钥，就是上面第4点的M2公钥，解密出签名中的H1
  4. 判断H2是否等于H1，即可验证签名是否真实。

最终，浏览器用M1公钥，加密随机成生的对称密钥S，给服务端；服务端用M1私钥解析后，用密钥S加密通信！

问题：对于中间人劫持

  • 证书无可伪造，其中有签名验证

  • 签名无法伪造，其中只有CA机构有私钥

  • 证书中M1公钥或部分信息也无法伪造，因为证书是完整的加密文件，有验证域名这些唯一信息等等

总结：

  • HTTPS的选择的方案是，数字证书（非对称+签名+对称）
  • 数据传输的加密方案不变，采用对称密钥
  • 用非对称+签名保证传输对称密钥不被挟持更改

# 申请证书

  • 现在申请数字证书，包括上面提到的CSR文件，各大网站基本帮助省略了。暂不介绍了。参考腾讯文档 (opens new window)
