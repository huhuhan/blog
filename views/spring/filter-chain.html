理解 FilterChain 过滤器链

寒铁 2021-01-21 spring 源码

# 概念前提

回顾下最早的Servlet方式编程中，Filter、Servlet等知识

# Filter & 链

  • 和Servlet类一样，由 Servlet 容器进行调用和执行的。
  • 一个Servlet可以注册到多个过滤器，一组过滤器，称为过滤器链
  • 过滤器的doFilter方法，最终调用FilterChain.doFilter才表示当前这个过滤器通过。
  • 过滤器调用FilterChain.doFilter后，按顺序执行下个过滤器，最后执行Servlet的service方法。

# FilterChain

  • 在简单Servlet编程中，FilterChain.doFilter的接口实现类是看不到的，因为都是Servel容器实现。
  • 在SpringBoot中内置了Tomcat依赖，所以可以看到其实现类。

# Spring MVC

Spring的Servlet应用就是Spring MVC，简化了Servlet、Filter在web.xml的配置和相互注册等等。

# ApplicationFilterChain

一个Spring Boot请求的过滤器拦截，只要内置是默认的Tomcat，就是从ApplicationFilterChain开始解析。

  • 这里引入了spring security的依赖包，就会多加载一个spring security的过滤器链

  • 这里会按顺序依次调用上面加载的过滤器链,，如下图。

    *注意每个都是过滤器链，内部又是一组过滤器调用。比如后面的Security过滤器链

# DelegatingFilterProxy

一个Servlet可以注册多个过滤器，Spring MVC的设计是注册一个委任的过滤器代理（也是过滤器）。Spring中的所有过滤器已经注入到IOC容器，可以根据委任的Bean名，动态调用过滤器。

# Spring Security 例子

根据上面得知，加入Spring Security的依赖后，ApplicationFilterChain中多了一组过滤链，其对应DelegatingFilterProxy的delegat对象就是org.springframework.security.webFilterChainProxy

# FilterChainProxy

  • 主要内容都在doFilterInternal中。其中getFilters将所有FilterChain的过滤器都解析出来，去重，再按顺序依次调用。
  • 这里this.filterChain的过滤器链可以通过WebSecurity的拦截配置。参考后续介绍。
  • 整理出的所有过滤器用VirtualFilterChain包装再调用，区分本身所属过滤链的调用冲突。

这里的出现的12、11、15的重复过滤器，是因为这个项目同时加了如下配置：

12个是AuthorizationServerConfigurerAdapter，即 认证服务配置

11个是ResourceServerConfigurerAdapter，即资源服务配置

15个是WebSecurityConfigurerAdapter，默认WebSecurity配置

ps：一般项目最多同时作为认证、资源服务。

# Spring Security Oauth2

引入Oauth2的依赖，默认过滤器链比Security的少4个，如图

  • 基于ResourceServerConfigurerAdapter来自定义配置权限拦截

  • 其中鉴权是OAuth2AuthenticationProcessingFilter就是token验证过滤器，基于TokenStore验证，

    可以扩展jwt等多种令牌方式

服务形式，分：授权服务器、认证服务器、资源服务器。

认证和授权，一般在同个服务，基于WebSecurityConfigurerAdapter（order=100）配置就行了。

资源服务器，即业务服务，基于ResourceServerConfigurerAdapter（order=3）配置。

若认证授权服务器中也当作资源服务器（比如也有业务接口，比如测试接口）基于ResourceServerConfigurerAdapter配置就行了，默认优先级高于WebSecurityConfigurerAdapter（配置了也不起效）。
