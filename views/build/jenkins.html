    # 推荐插件

    可以先统一下载需要的插件，一起重启Jenkins，否则每次下载插件再重启生效很费时间！！

    # 中文化

    插件名是：Localization: Chinese

    # GitLab

    • 插件名：GitLab (opens new window)

    • 会自动安装依赖的插件，比如Git、Gitlab Hook Plugin (opens new window)等等一推插件

    • 最常用的，就是在项目构建中，拉取项目仓库代码

      • 可以直接用http方式
      • 认证，直接用用户密码方式

    # Maven

    • 选择这个插件：Config File Provider (opens new window)

    • 【系统管理】-【全局工具配置】- 新增，保存后重启Jenkins才会安装

      安装后/var/jenkins_home/tools 下会有maven的目录，修改conf/setting.xml

      <!-- /var/jenkins_home 为dockers内部映射路径 -->
<localRepository>/var/jenkins_home/.m2/repository</localRepository>

    # NodeJS

    • 选择这个插件：NodeJS (opens new window)

    • 【系统管理】- 【全局工具配置】- 新增，保存后重启Jenkins才会安装

    # Publish Over SSH

    • 插件：Publish Over SSH (opens new window)
    • 【系统管理】-【系统配置】- Publish over SSH
    • 新增链接，部分字段需要点击【高级】
      • 昵称：name
      • 服务IP：hostname
      • 相对路径：Remote Directory，即包传送到服务器的【根目录】
      • 账户：username
      • 密码，需要先勾选Use password authentication, or use a different key
      • 端口：ssh端口，比如22
      • 超时：timeout
    • 测试连接

    • 实际应用，参考下面的【发布后端】等例子

    # 应用

    若是推荐插件安装，以下部分插件已经默认安装

    # 凭证管理

    【系统管理】-【凭证（Manage Credentials）】

    • 方式有：用户密码方式、API Token、私秘钥等等
    • 应用场景：gitlab仓库凭证、SSH凭证。填写时建议加上描述字段

    可参考博客 (opens new window)

    # 角色权限管理

    • 选择这个插件：Role-based Authorization Strategy (opens new window)，安装后重启

    • 【系统管理】-【全局安全配置】-【授权策略】选择Role-Based Strategy策略

    • 【系统管理】-【Manage and Assign Roles】

    • 【Manage Roles】创建角色，分配权限

      • 全局角色：控制全局权限，比如左侧菜单、视图等

        推荐一个子管理员角色和只读角色，如图

      • 项目角色：根据正则匹配，项目名称统一前缀，比如iot.*

        子管理员，直接勾选全部，如图iot，只读权限如图farm

    • 【Assgin Roles】分配用户-权限角色

      • 输入用户登录名添加（刷新后才会显示用户昵称）
      • 根据需求分配用户的权限角色

      两个用户，iot负责iot项目管理和iot-user只能构建iot项目

    • 【系统管理】-【读取配置】即重启jenkins，刷新配置信息

    # 新建任务

    • 选择【构建一个自由风格的软件项目】即可
    • 善于用最低下的【复制】功能，复制现有的项目任务，直接改配置

    # 发布后端

    1. 创建项目，比如【JavaA】，则项目路径是/workspace/JavaA

    2. 源码管理：拉取项目代码，基于插件Git

    3. 可选，构建触发器：【Build when a change is pushed to GitLab.】参考插件GitLab hook

    4. 构建：【调用顶层maven目录】基于插件Maven

      1. 选择版本

      2. 目标：即maven命令，参考下

        -DskipTests=true clean package
        1

      3. POM：默认项目根目录

        ${WORKSPACE}/pom.xml
        1

      4. 其他配置自行选择

    5. 构建：【Send files or execute commonds over SSH】即在对应服务器上编写执行脚本命令

      • 选择服务器，假设该服务器配置【根目录】是/yh/jenkins。参考插件publish-over-ssh

      • 当前【JavaA】的工作空间，即/workspace/JavaA

      • Source files：工作空间下的jar包路径。假设是target/javaA.jar

      • Remove prefix：去掉目录前缀，默认Source files的值。假设是target

      • Remote directory：上面的【根目录】拼接当前值。假设填入是/aa

      • 最终推送到服务器的jar包路径为：/yh/jenkins/aa/javaA.jar

      • 最终路径=【根目录】 + Remote directory +（Source files - Remove prefix）

      • Exec command：远程服务器的shell命令

    • 安装插件：（Gitlab Hook Plugin (opens new window)）、Gitlab Plugin (opens new window)、Build Authorization Token Root Plugin (opens new window)

    • Jekins：【系统管理】-【系统配置】找到Gitlab模块，添加配置，如下

      这一步骤似乎不需要！

      其中Credentials的凭证，类型选择GitLab API Token，这个token由Gitlab网站生成。

      GitLab：【用户设置】-【访问令牌】-【个人访问令牌】，权限都勾选，确认时效，生成。

    # 关于迁移

    方式一：不推荐

    复制所有数据，即目录/var/jenkins_home，全部拷贝至新服务器。

    实践测试，压缩后也有上百G至少，因为包括构建版本和依赖包仓库

    方式二：可选

    仅复制目录：/users、/plugins、/jobs。这样新服务器第一次初始化后，停止服务替换这些目录再重启服务。可保障用户权限信息、项目任务、插件不变

    但是！！Jenkins上的凭证信息都没有迁移过来。可能在其他地方！

    自定义安装的插件也需要重新安装，插件目录是/tools

    实践是重新配置了凭证，包括SSH和Git账户的凭证，项目需要更新凭证和SSH推送配置。

    这种方式有不太完善，后续有机会再测试更新
